Overview

项目概览

基于 LTCLAW-GY.X（个人 AI 智能体工作台）改造为游戏策划专属工具。核心目标：让 AI 自动维护项目索引，降低团队协作门槛，实现策划文档标准化产出。

核心定位
目标用户：小团队游戏策划（Unity 引擎项目）
数据同步：SVN
部署方式：本地部署，数据自主可控
核心需求：数值辅助 + 文档生成，通用化设计（什么项目都能用）
自动化边界：索引维护全自动，数值/文档决策由人类把控
Decisions

已拍板的设计决策

以下决策在讨论中已确认，是后续开发的硬约束。

决策 #1
自动化边界：索引全自动，其他人工决策
AI 全自动负责：SVN 变更检测、文件分类过滤、AI 解析生成 JSON 元数据、依赖图维护、registry.json 更新、commit 到 SVN、本地向量库重建、版本聚合报告
人类决策 AI 辅助：数值怎么调、加什么内容、改什么逻辑、提交数值表
索引层出错完全无害，大不了重跑一次
决策 #2
SVN 索引共享方案：.ltclaw_index/ 提交到 SVN
SVN 只存 AI 生成的文本元数据（JSON），不存向量。向量库在本地按需重建，秒级完成。索引结果写入 .ltclaw_index/ 后提交到 SVN，其他团队成员只需正常 svn update 同步。
决策 #3
SVN 变更感知：维护者本地轮询（非服务器 Hook）
无法直接 hook 到 SVN 服务器的 commit 事件。轮询服务运行在维护者本地机器上，定时执行 svn update 检测变更并触发索引重建。其他团队成员只需正常同步。不需要任何 SVN 服务器端配置。
决策 #4
防冲突策略：SVN 版本号做仲裁 + 随机抖动
每次准备提交前先 svn update 检查 last_synced_revision，已有人同步则跳过。commit 冲突则 svn update 拿到别人的结果。加随机抖动延迟减少无效竞争。建议先用"指定主索引机"模式（方案 A），后期按需升级为竞争模式（方案 B）。
决策 #5
Chat 改造：不新建页面，在现有 Chat 中扩展
数值查询结果在 PlanPanel Drawer（右侧抽屉）中展开，不在独立页面。Chat 就是 Chat，用户不会被踢出去。所有信息在 Chat 里原地展开和持久化。
决策 #6
决策按钮分两类位置：阻塞型在消息流，非阻塞型在 Drawer
阻塞型（消息流中）：写入操作确认，AI 在等，必须立即响应，复用现有 ApprovalCard
非阻塞型（Drawer 里）：[存入文档库] [确认调整] [标记有误] [忽略] [归档]，AI 不等，随时操作
决策 #7
文档模板开放配置，用户自己维护
模板不硬编码在系统里，而是用户自己维护（存 SVN Docs/Templates/），系统读取并遵守。界面上改的配置自动写回 project_config.yaml 并提交 SVN，团队所有人实时同步。
决策 #8
节点图库选择 reactflow，索引可视化分两期
几百张表需节点图库。推荐 reactflow（开发速度快，社区文档好）。列表视图先上线（够用），reactflow 节点图作为体验升级在主线稳定后单独排期。
决策 #9
版本记录分层存储：原始层(SVN) + 摘要层(30天) + 聚合层(周/里程碑)
change_history 只存"有意义的变更"（数值变化超过5%、行数变化超过1行、字段增删必须记录）。表级 JSON 保留最近 30 天明细，30 天前归档到 weekly/，weekly 保留最近 6 个月，6 个月前只保留 milestone 聚合。
决策 #10
不破坏现有功能，所有改动为扩展性质
现有页面/组件完全不动的文件约 95%。需改动的仅 4 个文件（constants.ts、Sidebar.tsx、PlanPanel/index.tsx、Chat/index.tsx），均为纯追加或扩展，不修改现有逻辑。

数值工作台

数值工作台有两个入口，服务不同场景：Chat 右侧 Drawer 用于实时查看和快速决策，独立页面 /game/workbench 用于深度调整和多表并列。

双入口设计
入口场景空间
Chat 右侧 Drawer 对话中实时查看引用字段，快速决策 380px 窄栏
独立页面 /game/workbench 专注深度调整，多表并列，全公式预览 全宽主内容区
关键设计
两个入口共享同一份数据，状态可以互相传递。从 Drawer 点击跳转到独立页面时携带上下文，自动滚动到对应列并高亮目标字段。

开发注意事项

7 条关键约束
#要点
1ResizablePanels 优先复用项目现有实现，不引入新依赖。若无，使用 useRef + mousemove 最小实现
2预览接口防抖 300ms，避免输入过程中频繁请求
3pendingChanges 存在组件本地 state 里。用户没点 [生成变更草稿] 前全部是本地状态，不请求写入
4多表并列超过4张时外层横向滚动，每列保持 240px 固定宽，不压缩
5Drawer 切换内容时只换内部渲染的组件，不重置 Drawer 的开关状态，不影响 Drawer 动画
6数值工作台是全新页面，不改动任何现有 Chat 逻辑。PlanPanel 里只加一个条件渲染，原分支完全不动
7AI 补全面板的"同类参考值"点击填入，只填入当前正在编辑（focus）的字段，无 focus 时 Toast 提示先选中字段

Technical

索引系统设计

索引是整个系统的基础。索引里不存表的每一行数据，只存表的"元数据"：字段定义 + 摘要 + 变更历史 + 依赖关系。

.ltclaw_index/ 文件结构

.ltclaw_index/ ├── registry.json # 中枢记录：版本号、文件清单、校验和 ├── project_config.yaml # 项目配置（团队共享） ├── dependency_graph.json # 全局依赖图（节点+边+置信度） ├── tables/ # AI 生成的表元数据 │ ├── SkillTable.json │ └── ItemTable.json ├── docs/ # AI 生成的文档索引 │ └── GDD.json ├── code/ # C# 轻量解析结果 └── history/ # 版本聚合 ├── weekly/ # 周聚合 └── milestone/ # 里程碑聚合

AI 自动维护完整流程

1. svn log 获取变更文件列表
|
2. 过滤文件（xlsx/csv/json/md/docx 通过，meta/png/fbx 忽略）
|
3. svn update 拉取最新内容
|
4. AI Indexer 按类型分发处理
|
数值表：解析表头+采样 → AI Prompt → JSON
文档：提取文本分块 → AI 生成摘要 → JSON
代码：提取类名/方法签名 → 轻量 JSON
|
5. 更新 dependency_graph.json（增量合并）
|
6. 更新 registry.json
|
7. svn commit .ltclaw_index/（防冲突机制）
|
8. 重建本地向量库（增量，只更新变化的文件）

防冲突流程

Step 1: svn update registry.json
读取 last_synced_revision = R_done
|
Step 2: svn info → HEAD = R_head
|
R_done >= R_head → 已有人同步，跳过
vs
R_done < R_head → 需要同步
|
Step 4-5: svn update + AI 处理 + 生成 JSON
|
commit 成功 → 完成
vs
commit 失败(conflict) → svn update 拿别人结果 → 跳过

文件类型分层处理策略

类型文件处理方式
数值类（高频） .xlsx / .csv / .json 解析表头+内容，存结构化缓存
文档类（低频） .md / .txt / .docx 分块 embedding，向量索引
代码类（可选） .cs（Unity 脚本） 提取类名/函数签名，不做全文索引
忽略类 .meta / .png / .fbx / .wav / Library/ / Temp/ 不处理

有意义的变更过滤规则

MEANINGFUL_CHANGE_THRESHOLD = { "numeric_delta": 0.05, # 数值变化超过5%才记录 "row_count_delta": 1, # 行数变化超过1行才记录 "field_change": True, # 字段增删必须记录 } # 过滤示例： # 格式调整、空格变化 → 过滤 # DamageCoeff 1.0→1.01 → 过滤（变化太小） # DamageCoeff 1.0→1.5 → 记录 # 新增一行数据 → 记录

Technical

安全边界

AI 可以做
起草数据、分配 ID（按 ID 段规划）、校验格式、生成 commit message

索引文件的 svn commit（出错无害，重跑就好）
AI 不应该做
直接 svn commit 数值表（索引文件可以，数值表不行）

原因：索引错了最多查询不准，数值表错了可能导致游戏崩溃或数值失衡
降低出错代价的工程手段
手段说明
写入前多重校验独立规则校验层（纯代码）：ID 区间、必填字段、外键引用、数值范围、字段类型
沙箱提交 + 自动回滚写影子副本，Unity 读影子副本跑校验，通过才覆盖正式副本
提交到专用分支AI 提交到 ltclaw-autowrite/ 分支，人工 review 后合并主干
变更规模限制单次 ≤10行、≤3字段、≤1张表，超过必须人工确认
不确定性暴露AI 遇到不确定的信息，标注出来让人填，不自己猜

Technical

依赖与技术栈

现有技术栈（不改动）
前端：React + TypeScript + Ant Design (bailianTheme) + Less
路由：React Router v6
状态：Context + Stores (Zustand/自定义)
后端：FastAPI + Python
主题：主色 #6B7280，已有 ThemeToggle（深色/浅色）
样式：全部继承 Ant Design，不引入新设计语言
唯一新增依赖
前端：reactflow（索引地图节点图）
后端：chromadb 或 faiss-cpu（本地向量库）
后端：openpyxl（Excel 解析，已在 full 依赖里）
系统：svn 命令行工具（subprocess 调用）

除节点图库外，没有任何重型新依赖，整体技术风险很低。
通用性边界：SVN 监听机制、AI 解析、自然语言查询、文档生成框架是通用的。每个项目需要一次性投入的是：project_config.yaml 初始配置、文档模板整理、已知依赖关系声明、ID 段划分声明。项目专属的设计原则和团队工作流永远需要人来定义和维护。

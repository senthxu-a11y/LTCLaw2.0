LTCLAW 游戏策划工作台 开发计划
优先级原则
这套系统的开发优先级按照能不能跑起来来排，而不是按功能完整度排。没有索引，查询就是空谈；没有查询，文档生成就没有上下文；没有文档生成，文档库就没有内容。所以每一层都是下一层的地基。

Code
P0  基础设施：没有它其他都跑不起来
P1  核心价值：直接解决最高频痛点
P2  提效工具：让体验更完整
P3  锦上添花：有了更好，没有也能用
P0 基础设施
这一阶段的目标只有一个：让索引稳定跑起来，全员同步。 其他所有功能都建立在索引之上，P0 不稳就不要往下走。

值得特别说明的是 SVN 轮询服务的设计。考虑到无法直接 hook 到 SVN 服务器的 commit 事件，轮询服务改为运行在维护者本地机器上，定时执行 svn update 检测变更并触发索引重建。索引结果写入 .ltclaw_index/ 后提交到 SVN，其他团队成员只需正常同步就能拿到最新索引，不需要任何 SVN 服务器端配置，也不依赖特定的服务器环境。

SVN 本地轮询服务	运行在维护者本地，定时检测变更触发索引；轮询间隔可在配置页调整	项目配置

P1 核心查询能力
索引有了之后，第一件要做的事是让策划能在 Chat 里问任何关于项目的问题，并且得到有依据的结构化回答，而不是 AI 凭空编造。这一阶段完成后，策划查字段含义、找相关表、看数值历史，都不需要再打开 Excel。

PlanPanel Drawer 的扩展也在这一阶段完成。查询过程中 AI 引用的所有表和字段会实时出现在右侧 Drawer 里，关键字段高亮显示，产出的草稿也会在 Drawer 里积累，随时可以操作，不打断对话流。

P1 知识库
知识库和查询能力属于同一优先级，因为两者互相依赖——查询质量取决于知识库的丰富程度，知识库的内容又主要来自查询过程中的沉淀。

这里最重要的设计是门槛要低。策划不需要填表单维护知识，只需要在 Chat 里自然地说出经验，系统自动识别并结构化存入。知识库也不是一次性建好的，随着使用不断积累，AI 的回答会越来越准。

P2 文档生成

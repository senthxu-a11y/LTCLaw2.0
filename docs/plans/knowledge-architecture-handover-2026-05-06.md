# LTClaw 策划知识库与工作台架构交接稿

> 生成时间：2026-05-06
> 适用范围：LTClaw 游戏策划工作台后续架构细化
> 目标：为后续 agent 提供一份可继续拆解实现的统一方案底稿，避免继续在同步、RAG、map、工作台写回之间来回摇摆。

---

## 0. 3 分钟速读摘要

如果后续 agent 时间有限，先只看这一节。

### 0.1 这轮架构的最终主线

系统已经不再走“每个人本地扫仓库、建索引、跑 RAG”的路线，而是改成：

1. 管理员是唯一的知识生产者。
2. 普通用户只消费已发布知识。
3. 数值工作台继续做业务编辑和引擎测试，不直接改正式知识库。
4. map 是必须存在的权威目录，来源是“LLM 初扫 + 人工修正”。
5. RAG 只负责说明和关系，精确值查询与精确修改必须走结构化数据链。

一句话概括：

管理员生产知识，普通用户消费知识，工作台编辑业务数据，RAG 负责说明与关系，结构化链负责精确读写，所有正式资产通过单独发布库统一分发。

### 0.2 角色分工

1. 管理员
   - 扫描资源
   - 生成和修正 map
   - 重建索引
   - 重建向量库
   - 发布正式版本

2. 普通用户
   - 只同步发布库
   - 只查已发布知识
   - 可用数值工作台快速改数、预览和测试
   - 不参与 map、索引、embedding

3. 工作台写入者
   - 可以查表、改表、上传给引擎测试
   - 改动进入测试方案或测试补丁，不自动进入知识库

### 0.3 系统分成三条链

1. 业务链
   - 数值工作台
   - 精确查询
   - 精确修改
   - 引擎测试

2. 知识治理链
   - 资源扫描
   - map 候选生成
   - 人工审核
   - 索引构建
   - 向量库构建
   - 发布

3. 知识消费链
   - 发布库同步
   - 知识查询
   - 文档/知识条目浏览
   - 来源路径定位

### 0.4 发布库是新的中心

所有正式知识资产统一放进一个单独的发布目录。

发布库包含：

1. manifest
2. 正式 map
3. 正式索引
4. 正式向量库
5. 发布日志

普通用户同步时：

1. 拉取发布库
2. 读取 manifest
3. 校验版本一致性
4. 原子切换到新版本

普通用户同步时绝不做：

1. 扫描仓库
2. 重建索引
3. embedding
4. 生成 map

### 0.5 RAG 和结构化链的边界

RAG 负责：

1. 找系统
2. 找表
3. 解释字段
4. 总结规则
5. 汇总文档说明
6. 找脚本证据

RAG 不负责：

1. 精确值查询
2. 直接改单元格
3. 行级事实判断

所以：

1. 说明和关系走 RAG
2. 精确值和修改走结构化查询与结构化写回

### 0.6 数值工作台改动和知识库的关系

数值工作台的主链路是快速测试，不走管理员接受：

1. 用户在工作台改数据
2. 生成测试方案或测试补丁
3. 预览影响
4. 导出或上传给引擎测试
5. 测完保留方案或丢弃方案

这些改动不会自动进入正式知识库。

管理员只在重建索引或发布知识版本时决定是否纳入某些已验证的测试方案：

1. 默认扫描本地项目正式资源。
2. 默认不纳入临时测试方案。
3. 管理员可在构建页选择少量已验证方案作为本次 release 输入。
4. 被纳入后才会影响下一次正式知识版本。

也就是说：

改数值是即时测试行为；是否成为正式知识，是管理员构建 release 时的选择，和普通用户的测试流程无关。

### 0.7 用户侧只看三种状态

为了减负，普通用户界面只建议暴露三种状态：

1. 已发布知识
2. 测试方案
3. 已废弃知识

同时，每条知识结果都显示：

1. 来源类型
2. 来源路径
3. 发布版本
4. 更新时间

### 0.8 后续 agent 优先干什么

如果后续要继续细化，建议优先顺序：

1. 发布库目录结构和 manifest
2. map 数据结构和审核 UX
3. 测试方案与 release 构建选择页的边界
4. 精确查询与精确修改 API
5. 最后再补 RAG 路由和向量库细节

原因：

先把治理边界、发布边界和数据边界定死，返工最少。

## 1. 这轮架构收敛后的总目标

本轮讨论后，系统目标已经从“每个人本地扫仓库、建索引、跑 RAG”收敛为：

1. 管理员是唯一的知识生产者。
2. 普通用户只消费已发布的知识资产。
3. 数值工作台继续承担快速查数、改数、预览和引擎测试辅助，不直接污染正式知识库。
4. map 是必须存在的权威目录，但它的生成方式是“LLM 初扫 + 人工修正”。
5. RAG 不是唯一查询手段，精确值查询与精确修改必须走结构化数据链。

一句话概括：

管理员负责生成和发布知识资产；普通用户只同步、查询和使用；工作台负责快速测试和结构化数值操作；RAG 负责说明和关系，不负责精确值读写。

## 2. 角色与权限模型

### 2.1 管理员

管理员是唯一允许参与知识治理的人，负责：

1. 扫描项目资源。
2. 运行 LLM 生成 v0 map 候选。
3. 人工修正 map。
4. 在构建 release 时选择哪些已验证测试方案可纳入正式知识体系。
5. 重建正式索引。
6. 重建正式向量库。
7. 发布正式版本到发布库。

管理员是唯一允许修改以下资产的人：

1. map。
2. 正式索引。
3. 正式向量库。
4. 发布版本与发布清单。

### 2.2 普通用户

普通用户明确不参与知识生产，能力边界如下：

1. 不参与 map 生成。
2. 不参与索引重建。
3. 不参与 embedding。
4. 只手动同步发布库。
5. 只查询已发布知识。
6. 只读查看知识条目、系统信息和来源路径。
7. 正常使用数值工作台进行查数、改数、预览和引擎测试。

### 2.3 工作台写入者

工作台写入能力与知识治理能力不是同一条权限轴。

结论：

1. 能改数值工作台，不等于能改 map。
2. 能改数值工作台，不等于能发布知识库。
3. 能保存测试方案，不等于该方案会进入正式知识库。
4. 后续权限设计至少要拆成三轴：
   - 知识治理权限
   - 数值工作台权限
   - 只读查询权限

## 3. 系统拆成三条链

### 3.1 业务链

业务链负责：

1. 数值工作台编辑。
2. 精确查询表数据。
3. 精确修改表数据。
4. 将改动上传给引擎测试。
5. 保存、复制、丢弃测试方案。

业务链特点：

1. 面向效率。
2. 可以写。
3. 不自动更新 map。
4. 不自动更新正式索引。
5. 不自动更新向量库。
6. 不要求管理员接受才能测试。

### 3.2 知识治理链

知识治理链负责：

1. 扫描表、文档、脚本资源。
2. 生成候选 map。
3. 人工修正 map。
4. 在构建 release 时选择是否纳入已验证测试方案。
5. 构建正式索引。
6. 构建正式向量库。
7. 发布正式版本。

知识治理链特点：

1. 只有管理员可操作。
2. 是唯一的知识生产入口。
3. 所有对外知识变化必须通过这里。

### 3.3 知识消费链

知识消费链负责：

1. 普通用户手动同步发布库。
2. 查询正式知识。
3. 浏览系统知识条目。
4. 查看来源路径。
5. 使用知识结果辅助策划工作。

知识消费链特点：

1. 只读。
2. 只消费正式发布物。
3. 不参与后台自动建库。

---

## 4. 发布库是新的中心

所有正式知识资产统一放入一个单独目录，后续 UX 需要围绕这个“发布库”展开可视化管理。

### 4.1 发布库的职责

发布库承担以下职责：

1. 承载正式版 map。
2. 承载正式版结构化索引。
3. 承载正式版向量库。
4. 承载发布清单与发布日志。
5. 作为普通用户同步的唯一来源。

### 4.2 发布库建议内容

至少包含：

1. 发布清单 manifest
   - 发布版本号
   - 发布时间
   - 发布人
   - map 版本
   - 表索引版本
   - 文档索引版本
   - 脚本证据索引版本
   - 向量库版本

2. 正式 map

3. 正式结构化索引
   - 表事实索引
   - 文档知识条目索引
   - 脚本证据索引

4. 正式向量库

5. 发布日志

### 4.3 发布库不应包含的内容

1. LLM 扫描候选结果。
2. 管理员未审核的候选 map。
3. 普通用户本地生成物。
4. 工作台临时测试改动。

### 4.4 普通用户同步行为

普通用户同步不再意味着“扫描仓库并建库”，而是固定动作：

1. 拉取发布库。
2. 读取 manifest。
3. 校验 map、索引、向量库版本一致性。
4. 原子切换到新版本。
5. 刷新知识库浏览和查询结果。

普通用户同步时禁止发生：

1. 候选扫描。
2. map 生成。
3. 索引重建。
4. embedding。

## 5. map 的定位与生成流程

### 5.1 map 是第一公民

map 不再被视为普通文档，而是整个知识体系的权威目录。

它负责：

1. 定义系统边界。
2. 定义正式表范围。
3. 定义文档归属。
4. 定义脚本证据域。
5. 定义废弃资源。
6. 决定后续检索路由。

### 5.2 map 的来源

map 的来源已明确为：

1. LLM 初扫生成 v0 候选。
2. 管理员人工修正。
3. 发布为正式 map。
4. 后续新增资源继续走“候选 -> 修正 -> 发布”。

### 5.3 map 至少要覆盖的对象

1. 系统
2. 表
3. 文档
4. 脚本域
5. 公式主题
6. 状态
7. 关系

关系至少包括：

1. 系统包含哪些表。
2. 系统包含哪些文档。
3. 系统关联哪些脚本域。
4. 表被哪些脚本消费。
5. 文档说明哪些系统或表。
6. 哪些对象已废弃。

### 5.4 map 的 UX 方向

不要让管理员直接手改大 YAML 或大 JSON。

建议做成三块：

1. 候选收件箱
   - 展示 LLM 扫出来的新表、新文档、新脚本域和候选归属
   - 管理员操作：接受、改归属、标废弃、忽略

2. 系统地图页
   - 按系统展示正式表、正式知识条目、脚本域、状态、最近变更

3. 发布页
   - 展示待发布变更、影响系统、是否重建索引、是否生成新向量库

## 6. 索引体系的拆分

不再做一个混合大库，而是拆成多个层次。

### 6.1 表结构索引

当前已经接近该层，主要承载：

1. 表名
2. 字段
3. 主键
4. 行数
5. 系统归属
6. 表摘要
7. 依赖关系

用途：

1. 系统理解
2. 表导航
3. 字段解释
4. 结构级问答

### 6.2 表事实索引

这是后续必须补强的层，承载：

1. 行级事实
2. 主键值
3. 关键字段值
4. 精确数值
5. 行状态

用途：

1. 精确查询
2. 精确定位
3. 精确修改后的验证

### 6.3 文档知识索引

承载：

1. 已批准文档的知识条目
2. 文档摘要
3. 关联系统
4. 关联表
5. 来源路径

用途：

1. 业务说明
2. 规则背景
3. 文档型问答

### 6.4 脚本证据索引

承载：

1. 脚本符号
2. 配置消费点
3. 表字段引用
4. 功能入口

用途：

1. 实现验证
2. 影响分析
3. 代码证据补强

### 6.5 向量库的边界

向量库只承载：

1. 文档说明
2. 表结构语义
3. 脚本证据语义

向量库不承载：

1. 全量精确表数据事实
2. 所有行级值
3. 所有数值变化历史

结论：向量库不能单独承担“精确数据查询”。

## 7. RAG 的职责与边界

### 7.1 RAG 负责什么

RAG 适合处理：

1. 找系统。
2. 找表。
3. 解释字段含义。
4. 总结规则。
5. 汇总文档说明。
6. 找相关脚本证据。

### 7.2 RAG 不负责什么

RAG 不应承担：

1. 精确值查询。
2. 直接改单元格。
3. 直接做行级事实判断。
4. 代替结构化数据库式查询。

### 7.3 最终分工

1. “说明和关系”走 RAG。
2. “精确值和修改”走结构化查询与结构化写回。

## 8. 数值工作台的定位

数值工作台不改其“快速测试与结构化数值操作”定位，但要与知识库主链解耦。

### 8.1 数值工作台继续负责

1. 查表。
2. 改测试数值。
3. 预览影响。
4. 辅助引擎测试。
5. 保存、复制、导出或丢弃测试方案。

### 8.2 数值工作台不直接负责

1. 改 map。
2. 改正式索引。
3. 改正式向量库。
4. 即时发布知识。

### 8.3 工作台改动与知识库的关系

工作台改动默认只是测试方案或测试补丁，不自动进入知识库，也不需要管理员接受才能测试。

正确链路：

1. 用户在工作台精确修改数据。
2. 系统生成结构化测试 patch。
3. 用户预览影响。
4. 用户导出或上传给引擎测试。
5. 用户选择保留为测试方案或丢弃。
6. 如测试方案已验证，可被标记为发布候选。
7. 管理员只在构建 release 时选择是否纳入这些发布候选。

结论：工作台改动不是知识审批流，而是快速测试流；是否成为正式知识，只在 release 构建时处理。

## 9. 精确查询与精确修改的架构方向

如果要做到“直接查询”和“精确修改数据”，不能依赖向量库，必须建立独立结构化数据链。

### 9.1 精确查询

应该支持：

1. 按表查询。
2. 按主键查询。
3. 按字段过滤。
4. 按精确定位查询。

返回应是结构化结果，而不是自然语言摘要。

### 9.2 精确修改

应当收敛成有限几类操作：

1. 修改单元格。
2. 新增行。
3. 删除行。
4. 后续再支持新增列与废弃列。

### 9.3 修改的内部表达

每次修改都要变成结构化 patch，而不是让模型自由改文本。

patch 至少包含：

1. 操作类型
2. 目标表
3. 主键或定位条件
4. 目标字段
5. before
6. after
7. 来源路径

### 9.4 修改后的流转流程

1. 修改进入测试方案或测试 patch。
2. 用户直接用于预览、导出或引擎测试。
3. 普通测试方案默认不进入正式知识。
4. 已验证方案可被标记为发布候选。
5. 管理员在构建 release 时选择是否纳入发布候选。
6. 纳入后由发布流程统一重建索引和向量库。

## 10. 表改动场景的覆盖结论

已经讨论过的结论如下：

### 10.1 当前设计可相对覆盖

1. 新增新表。
2. 新增新列。
3. 废弃部分列的结构变化。

前提：管理员重新确认 map 并重新发布。

### 10.2 当前设计只能弱覆盖

1. 增加行。
2. 删除部分行。

原因：现有表索引更偏结构级，不是行事实级。

### 10.3 当前设计天然覆盖不足

1. 修改精确数值。

原因：向量与表结构检索不会天然追踪逐行逐值变化。

### 10.4 结论

后续必须补一层“表事实索引”，不能只依赖当前表结构索引与向量库。

## 11. 同步策略的收缩

### 11.1 SVN 同步

当前建议已经明确：

1. 自动 SVN 同步退出主流程。
2. 先改成只能手动同步。
3. 普通用户的唯一同步动作是“同步发布库”。

原因：

1. 自动轮询带来后台不确定性。
2. 容易把“拉 SVN、建索引、刷新知识”混成一个动作。
3. 与 map 审核和发布态知识不一致。

### 11.2 heartbeat

heartbeat 被判断为与当前策划知识主链解耦。

建议：

1. 默认关闭。
2. 不参与 game workbench 主流程。
3. 不参与自动同步、自动索引或自动问答。

## 12. 用户侧状态表达

为了减负，普通用户不应看到太多内部术语，只看直观状态。

建议统一成：

1. 已发布
2. 测试方案
3. 已废弃

避免直接暴露：

1. 实验数据
2. 草稿知识
3. 临时索引
4. embedding 状态
5. 管理员构建 release 的内部选择

同时，每条知识结果固定显示：

1. 来源类型：表 / 文档 / 脚本
2. 来源路径
3. 发布版本
4. 更新时间

目的：帮助用户快速锁定源文件。

## 14. 这轮架构拍板后的关键规则

当前已基本拍板的规则如下：

1. 普通用户不参与 map、索引、embedding。
2. 所有正式知识成果放入单独发布库。
3. map 是必须的，并由“LLM 初扫 + 人工修正”生成。
4. 数值工作台改动默认不进入向量库或正式知识库。
5. 工作台改动先进入测试方案或测试 patch。
6. 普通测试不需要管理员接受。
7. 只有管理员能在构建 release 时选择已验证发布候选是否纳入正式知识体系。
8. 纳入知识后必须走统一发布，不做即时局部入库。
9. 普通用户只消费已发布版本。
10. 向量库不承担精确值查询。
11. 所有知识结果必须显示来源路径。

### 17.2 核心模块职责

#### A. 资源发现模块

输入：

1. 表根目录
2. 文档根目录
3. 脚本根目录
4. 本地项目目录

输出：

1. 候选资源清单
2. 新增/删除/变更资源摘要
3. 供 LLM 初扫使用的资源元数据

#### B. map 生成与审核模块

输入：

1. 候选资源清单
2. LLM 输出的 v0 map 候选
3. 管理员修正结果

输出：

1. 正式 map
2. 待发布 map 变更

#### C. 索引构建模块

输入：

1. 正式 map
2. 已批准的表、文档、脚本域
3. 构建 release 时被管理员选择纳入的发布候选

输出：

1. 表结构索引
2. 表事实索引
3. 文档知识索引
4. 脚本证据索引

#### D. 向量库构建模块

输入：

1. 已批准文档知识条目
2. 表结构语义条目
3. 脚本证据语义条目

输出：

1. 正式向量库

限制：

1. 不直接存全量精确表数据
2. 不直接存所有逐行逐值事实

#### E. 发布模块

输入：

1. 正式 map
2. 正式索引
3. 正式向量库
4. 发布说明

输出：

1. manifest
2. 新发布版本目录
3. 发布日志

#### F. 消费模块

输入：

1. 发布库版本

输出：

1. 普通用户本地已加载版本
2. 知识查询结果
3. 来源定位信息

#### G. 工作台结构化读写模块

输入：

1. 指定表
2. 指定主键
3. 指定字段
4. 指定 patch

输出：

1. 精确查询结果
2. 精确修改结果
3. 测试方案或测试 patch

### 17.3 数据流一：知识生产流

```text
项目原始资源
   -> 资源扫描
   -> LLM 初扫生成 v0 map
   -> 管理员审核修正
   -> 正式 map
   -> 按 map 构建正式索引
   -> 构建正式向量库
   -> 发布到发布库
```

### 17.4 数据流二：普通用户消费流

```text
普通用户点击“同步发布库”
   -> 拉取发布库
   -> 读取 manifest
   -> 校验 map/索引/向量库版本一致性
   -> 原子切换当前版本
   -> 刷新知识查询和浏览视图
```

### 17.5 数据流三：工作台业务改动流

```text
用户在数值工作台改表
   -> 形成结构化 patch
   -> 预览影响
   -> 导出或上传给引擎测试
   -> 保留为测试方案或丢弃
   -> 已验证方案可标记为发布候选
   -> 管理员在 release 构建时选择是否纳入
```

### 17.6 数据流四：精确查询流

```text
用户提出查询
   -> 如果是说明/关系问题：走 map 路由 + RAG
   -> 如果是精确值问题：走表事实索引/结构化查询
   -> 返回结果时统一显示来源路径与发布版本
```

### 17.7 原子切换原则

普通用户加载发布版本时，必须保证以下资产始终同版本：

1. map
2. 表结构索引
3. 表事实索引
4. 文档知识索引
5. 脚本证据索引
6. 向量库

禁止出现：

1. 新 map + 旧向量库
2. 新表事实索引 + 旧文档知识索引
3. 新 manifest + 半切换状态

### 18.1 权限维度

后续至少拆成三条权限轴：

1. 知识治理权限
2. 数值工作台权限
3. 只读查询权限

### 18.2 角色矩阵

| 能力 | 管理员 | 普通用户 | 工作台写入者 |
|---|---|---|---|
| 扫描项目资源 | 是 | 否 | 否 |
| 运行 LLM 生成 map 候选 | 是 | 否 | 否 |
| 修改 map | 是 | 否 | 否 |
| 构建时选择发布候选 | 是 | 否 | 否 |
| 重建正式索引 | 是 | 否 | 否 |
| 重建向量库 | 是 | 否 | 否 |
| 发布版本 | 是 | 否 | 否 |
| 同步发布库 | 是 | 是 | 是 |
| 查询已发布知识 | 是 | 是 | 是 |
| 查看来源路径 | 是 | 是 | 是 |
| 精确查询表数据 | 是 | 是 | 是 |
| 精确修改表数据 | 可选 | 可选 | 是 |
| 上传改动用于引擎测试 | 可选 | 可选 | 是 |
| 保存/丢弃测试方案 | 可选 | 可选 | 是 |
| 使发布候选进入正式知识 | 是 | 否 | 否 |

### 18.3 关键权限结论

1. “能改工作台”不等于“能改知识库”。
2. “能查知识库”不等于“能发布知识库”。
3. “能保存测试方案”不等于“方案会进入正式知识”。
4. 普通用户始终只能消费正式发布物。

## 19. 页面结构与 UX 操作流

### 19.1 页面结构总览

后续推荐至少有 6 个核心页面或视图。

#### 普通用户侧

1. 发布库同步页
2. 知识库查询页
3. 文档/知识条目浏览页
4. 数值工作台页

#### 管理员侧

5. map 审核页
6. 发布管理页

如后续需要，还可补：

1. 候选资源扫描页
2. 发布候选选择页

### 19.2 普通用户页一：发布库同步页

#### 目标

让普通用户一眼知道：

1. 当前使用的是哪个发布版本
2. 是否有可更新版本
3. 上次同步是否成功
4. 同步后刷新了哪些知识资产

#### 页面核心信息

1. 当前已加载版本
2. 最新可用版本
3. 上次同步时间
4. 本次更新摘要
    - map 是否更新
    - 索引是否更新
    - 向量库是否更新

#### 核心按钮

1. 同步发布库

#### 点击后的固定动作

1. 拉取发布库
2. 读取 manifest
3. 校验版本一致性
4. 原子切换
5. 刷新数据视图

#### 明确不做的事

1. 扫描仓库
2. 重建索引
3. 重新 embedding
4. 自动改 map

### 19.3 普通用户页二：知识库查询页

#### 目标

支持用户快速获取：

1. 系统说明
2. 表与字段解释
3. 文档知识条目
4. 脚本证据摘要
5. 来源路径

#### 搜索结果必须显示

1. 标题
2. 类型：表 / 字段 / 文档 / 脚本
3. 摘要
4. 来源路径
5. 发布版本
6. 更新时间

#### 查询分流原则

1. 说明和关系问题：优先走 RAG
2. 精确值问题：引导切到结构化查询或工作台

#### UX 关键点

搜索结果中应明确提示：

1. 该结果是否为“已发布”
2. 该结果来自哪个文件
3. 如果是说明性结果，不要伪装成精确值结果

### 19.4 普通用户页三：文档/知识条目浏览页

#### 目标

不是给普通用户看扫描草稿，而是浏览已发布知识条目。

#### 建议展示方式

1. 按系统分组
2. 按文档主题分组
3. 每条条目提供：
    - 标题
    - 摘要
    - 关联表
    - 来源路径
    - 发布版本

#### 结论

第一阶段可以先不提供原文镜像阅读器，只提供知识化结果和来源路径。

### 19.5 普通用户页四：数值工作台页

#### 定位

数值工作台保留为快速测试与结构化数值操作区，不改其核心职责。

#### 第一阶段能力

1. 精确查询表
2. 按主键定位行
3. 修改单元格
4. 预览影响
5. 导出或上传给引擎测试
6. 保存、复制或丢弃测试方案

#### UI 必须明确的状态

1. 当前发布：来自正式知识版本或正式资源
2. 测试方案：用户临时实验，可保留或丢弃
3. 发布候选：已验证并可在下次 release 构建时被管理员选择
4. 已废弃：历史保留对象

#### 关键提醒

工作台页面必须告诉用户：

1. 当前修改不会自动进入知识库
2. 测试不需要管理员接受
3. 只有被标记为发布候选，并在 release 构建时被管理员选择，才会影响下一次知识版本

### 19.6 管理员页一：map 审核页

#### 目标

低成本地修正 LLM 生成的 map 候选，而不是直接改文件。

#### 页面组成

1. 候选收件箱
    - 新表
    - 新文档
    - 新脚本域
    - 疑似废弃资源

2. 系统地图主视图
    - 每个系统下的正式表
    - 正式知识条目
    - 脚本域
    - 状态

3. 审核动作
    - 接受
    - 改归属
    - 标废弃
    - 忽略

#### UX 关键点

管理员修改的不是“文件文本”，而是“系统关系”。

### 19.7 管理员页二：发布管理页

#### 目标

把候选变更、map、索引、向量库的发布动作收敛成一个统一入口。

#### 页面需展示

1. 待发布版本号
2. 本次变更涉及的系统
3. map 是否变化
4. 哪些索引要重建
5. 可选发布候选列表
6. 是否重建向量库
7. 发布备注

#### 核心按钮

1. 生成发布包
2. 发布正式版本

#### 发布原则

1. 先做整库发布，保证简单稳定
2. 后续再考虑优化成增量发布

### 19.8 发布候选选择页（可选但建议）

#### 目标

承接已验证测试方案与知识 release 构建之间的桥梁。

#### 展示内容

1. 哪张表被改了
2. 哪些主键被改了
3. 哪些字段变了
4. before / after
5. 引擎测试结果或说明
6. 是否建议纳入本次 release

#### 管理员动作

1. 本次 release 纳入
2. 本次 release 不纳入
3. 暂缓到后续 release
